## 1. **Implement Two-Factor Authentication (2FA) for MariaDB:**
**Objective:**
Secure MariaDB user authentication with Two-Factor Authentication.

#### **Step 1: Install Required Packages**
Ensure you have the necessary packages installed:

```bash
sudo apt update
sudo apt install libpam-google-authenticator
```

#### **Step 2: Configure PAM for Two-Factor Authentication**
Edit the PAM configuration file for MariaDB:
```bash
sudo nano /etc/pam.d/mysql
```

Add the following line at the end of the file:
```bash
auth required pam_google_authenticator.so
```
This instructs PAM to use Google Authenticator for MariaDB authentication.

#### **Step 3: Restart MariaDB Service**
Restart the MariaDB service to apply the PAM changes:
```bash
sudo systemctl restart mariadb
```

#### **Step 4: Configure Google Authenticator for MariaDB Users**
Switch to the MariaDB user you want to enable 2FA for:
```bash
sudo mysql -u root -p
```
Run the following SQL query to enable 2FA for the user:

```sql
SET PASSWORD FOR 'your_user'@'localhost' = PASSWORD('*2FA*YourPassword');
```

Replace `'your_user'` with the MariaDB user you want to secure and choose a secure password for the user.

#### **Step 5: Test Two-Factor Authentication**

Attempt to log in to MariaDB as the user you configured for 2FA. You should be prompted for both your MariaDB password and the 2FA code generated by the Google Authenticator app.

#### **Conclusion:**
By implementing Two-Factor Authentication for MariaDB on Debian, you enhance the security of user authentication. Users must provide both their MariaDB password and a time-sensitive code from their 2FA app to access the MariaDB server, adding an extra layer of protection against unauthorized access.

Note: Ensure to test the setup thoroughly and have backup access methods in case any issues arise with the 2FA setup.

## 2. **Regularly Update and Patch MariaDB:**
**Objective:**
Set up a routine to regularly update and patch MariaDB for enhanced security on your Debian system.

#### **Step 1: Update Package Repositories**

Before updating MariaDB, ensure your package repositories are up-to-date:

```bash
sudo apt update
```

#### **Step 2: Check MariaDB Version**

Check your current MariaDB version to determine if updates are available:

```bash
sudo mysql -V
```

#### **Step 3: Upgrade MariaDB**

If a newer version is available, upgrade MariaDB to the latest version:

```bash
sudo apt upgrade mariadb-server
```

#### **Step 4: Confirm MariaDB Version**

After the upgrade, verify that MariaDB has been updated to the latest version:

```bash
sudo mysql -V
```

#### **Step 5: Set Up Automated Updates (Optional)**

Consider setting up unattended-upgrades to automate the update process:

```bash
sudo apt install unattended-upgrades
sudo dpkg-reconfigure unattended-upgrades
```

Follow the prompts to configure automatic updates.

#### **Step 6: Schedule Regular Updates**

Create a scheduled task to regularly check for updates and install them. Open your crontab file:

```bash
sudo crontab -e
```

Add the following line to schedule daily updates at a convenient time:

```bash
0 2 * * * apt update && apt upgrade -y
```

This example schedules updates at 2 AM daily. Adjust the timing to suit your server's usage patterns.

#### **Conclusion:**
By regularly updating and patching MariaDB, you ensure that your database server benefits from the latest security enhancements and bug fixes. This proactive approach helps safeguard your system against potential vulnerabilities.

Note: Always test updates in a safe environment before applying them to a production server. Automated updates can be useful, but exercise caution to prevent unexpected issues.


## 3. **Monitor MariaDB Logs**

**Objective:**
Establish a system to monitor MariaDB logs for enhanced security and performance insights on your Debian system.

#### **Step 1: Enable MariaDB Error Logging**

Edit the MariaDB configuration file to enable error logging:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Add the following lines in the `[mysqld]` section:

```ini
[mysqld]
log_error = /var/log/mysql/error.log
```

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Step 2: View MariaDB Error Logs**

Access the MariaDB error logs to monitor for any issues:

```bash
sudo tail -f /var/log/mysql/error.log
```

This command displays the last few lines of the error log and updates in real-time. Monitor this log for any errors or warnings related to MariaDB.

#### **Step 3: Enable General Query Logging (Optional but Recommended)**

You can also enable general query logging for a more comprehensive overview of MariaDB activity. Edit the MariaDB configuration file again:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Add the following lines to enable general query logging:

```ini
[mysqld]
general_log = 1
general_log_file = /var/log/mysql/general.log
```

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Step 4: View General Query Logs**

Access the general query logs to monitor MariaDB activity:

```bash
sudo tail -f /var/log/mysql/general.log
```

This provides insights into the SQL statements executed on the MariaDB server.

#### **Conclusion:**
Monitoring MariaDB logs on Debian allows you to proactively detect and address potential issues. Regularly reviewing error logs and, optionally, general query logs provides valuable information for troubleshooting and optimizing database performance.

Note: Be cautious with general query logging in a production environment as it can generate substantial log data. Adjust log levels and retention based on your monitoring needs and server resources.


## 4. **Implement Database Auditing**

**Objective:**
Set up MariaDB to log and audit database activities for enhanced security and compliance on your Debian system.

#### **Step 1: Enable MariaDB Audit Plugin**

Check if the `audit` plugin is already installed:

```bash
sudo mysql -u root -p -e "SHOW PLUGINS;"
```

If the `audit` plugin is not present, install it:

```bash
sudo mysql -u root -p -e "INSTALL PLUGIN audit SONAME 'audit_log';"
```

#### **Step 2: Configure MariaDB Audit Plugin**

Create a configuration file for the MariaDB audit plugin:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Add the following lines to configure the audit plugin:

```ini
[mysqld]
plugin-load-add = audit_log.so
audit_log_format = JSON
audit_log_rotate_on_size = 1000000
```

This configures the audit plugin to log in JSON format and rotate logs when they reach 1 MB.

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Step 3: Verify Audit Plugin Status**

Check the status of the audit plugin to ensure it's active:

```bash
sudo mysql -u root -p -e "SHOW PLUGINS;"
```

Look for the `AUDIT` plugin in the list.

#### **Step 4: Review Audit Logs**

Access the audit logs to review MariaDB activity:

```bash
sudo cat /var/log/mysql/audit.log
```

This command displays the contents of the audit log. Monitor this log for a comprehensive record of MariaDB events.

#### **Conclusion:**
Enabling the MariaDB audit plugin on Debian provides a detailed log of database activities, helping you identify potential security threats, track changes, and comply with regulatory requirements. Regularly review the audit logs for insights into your database's health and security.

Note: Adjust audit log configuration settings based on your monitoring needs and server resources. Periodically archive or rotate audit logs to manage storage space effectively.

## 5. **Restrict Network Access**

**Objective:**
Enhance the security of MariaDB by configuring network access restrictions on your Debian system.

#### **Step 1: Identify MariaDB Binding Address**

Check the current binding address of MariaDB:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Look for the `bind-address` parameter. It should be set to `127.0.0.1` by default, restricting MariaDB to localhost.

#### **Step 2: Allow Specific IP Addresses (Optional)**

If you need to allow specific IP addresses, you can modify the `bind-address` parameter. For example, to allow connections from IP address `192.168.1.2`, update the configuration file:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

```ini
[mysqld]
bind-address = 192.168.1.2
```

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Step 3: Configure Firewall Rules**

Ensure that your firewall only allows necessary connections to the MariaDB port (default is 3306). If using UFW, you can allow a specific IP or subnet:

```bash
sudo ufw allow from 192.168.1.2 to any port 3306
```

Replace `192.168.1.2` with the IP address you want to allow.

#### **Step 4: Deny External Connections (Optional)**

To completely deny external connections, set the `bind-address` to `127.0.0.1`:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

```ini
[mysqld]
bind-address = 127.0.0.1
```

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Conclusion:**
By restricting network access, you minimize the exposure of your MariaDB server to potential threats. Only allow connections from trusted IP addresses, reducing the risk of unauthorized access and improving overall database security.

Note: When implementing network restrictions, ensure that you don't accidentally lock yourself out. Test the changes in a controlled environment before applying them to a production server.

## 6. **Regularly Back Up Your MariaDB**

**Objective:**
Establish a routine for regular backups to ensure data integrity and facilitate recovery on your Debian system.

#### **Step 1: Choose a Backup Method**

There are various ways to back up a MariaDB database, such as using `mysqldump` for logical backups or setting up a backup tool like `Mariabackup` for physical backups. In this example, we'll use `mysqldump`.

#### **Step 2: Create a Backup Script**

Create a backup script that uses `mysqldump` to export the database to a file:

```bash
sudo nano /usr/local/bin/backup_mariadb.sh
```

Add the following script, adjusting the parameters as needed:

```bash
#!/bin/bash

# Database credentials
user="your_db_user"
password="your_db_password"
host="localhost"
db_name="your_database"

# Backup directory
backup_dir="/path/to/backup"

# Date format for the backup file
date=$(date +"%Y%m%d_%H%M%S")

# Dump database to SQL file
mysqldump --user=$user --password=$password --host=$host $db_name > $backup_dir/$db_name_$date.sql

# Compress the backup file
gzip $backup_dir/$db_name_$date.sql

# Remove backups older than 7 days
find $backup_dir -name "$db_name*.gz" -type f -mtime +7 -exec rm {} \;
```

Save the file and make it executable:

```bash
sudo chmod +x /usr/local/bin/backup_mariadb.sh
```

#### **Step 3: Schedule Regular Backups with Cron**

Open the crontab file to schedule the backup script:

```bash
sudo crontab -e
```

Add the following line to schedule a daily backup at 2 AM:

```bash
0 2 * * * /usr/local/bin/backup_mariadb.sh
```

Save the file.

#### **Conclusion:**
Regularly backing up your MariaDB database ensures that you can quickly recover from data loss or system failures. Adjust the backup script and schedule based on your database size, update frequency, and storage constraints.

Note: Test the backup and restore processes to ensure they work as expected. Store backup files in a secure location and consider offsite storage for additional protection.

## 7. **Implement Network Encryption**

**Objective:**
Enhance the security of MariaDB by encrypting network communication to protect data in transit on your Debian system.

#### **Step 1: Generate SSL Certificates**

Create SSL certificates for MariaDB:

```bash
sudo mkdir /etc/mysql/ssl
sudo openssl req -newkey rsa:2048 -days 365 -nodes -x509 -keyout /etc/mysql/ssl/server-key.pem -out /etc/mysql/ssl/server-cert.pem
```

Follow the prompts to provide information for the certificate.

#### **Step 2: Secure SSL Certificates**

Set secure permissions for the SSL certificates:

```bash
sudo chmod 600 /etc/mysql/ssl/server-*
sudo chown mysql:mysql /etc/mysql/ssl/server-*
```

#### **Step 3: Configure MariaDB for SSL**

Edit the MariaDB configuration file to enable SSL:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Add the following lines:

```ini
[mysqld]
ssl-ca=/etc/mysql/ssl/server-cert.pem
ssl-cert=/etc/mysql/ssl/server-cert.pem
ssl-key=/etc/mysql/ssl/server-key.pem
```

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Step 4: Verify SSL Encryption**

Connect to MariaDB and verify SSL encryption:

```bash
mysql -u your_user -p --host=127.0.0.1 --port=3306 --ssl-ca=/etc/mysql/ssl/server-cert.pem
```

Replace `your_user` with your MariaDB username. You should be prompted to accept the SSL certificate.

#### **Conclusion:**
Implementing network encryption for MariaDB ensures that data transmitted between the database server and clients is secure and protected from unauthorized access. Regularly update SSL certificates and monitor for any security advisories related to SSL/TLS.

Note: Test the SSL configuration thoroughly to ensure proper encryption. Adjust configurations based on your security requirements and compliance standards.

## 8. **Use Appropriate MariaDB Storage Engines**

**Objective:**
Optimize MariaDB performance by selecting and utilizing appropriate storage engines for your Debian system.

#### **Step 1: Understand MariaDB Storage Engines**

MariaDB supports various storage engines, each designed for specific use cases. Common engines include InnoDB, MyISAM, and Aria. Research and understand the characteristics of each engine to determine which is best suited for your application.

#### **Step 2: Selecting a Storage Engine**

Consider the following scenarios when selecting a storage engine:

- **InnoDB:** Use for transactions, foreign keys, and ACID compliance. Suitable for applications with frequent updates and relational data.

- **MyISAM:** Suitable for read-heavy applications. Does not support transactions or foreign keys but is fast for read operations.

- **Aria:** A replacement for MyISAM with improved performance. It supports transactions and is suitable for read and write-intensive applications.

#### **Step 3: Configure Default Storage Engine**

Edit the MariaDB configuration file to set the default storage engine. Open the configuration file:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Add the following line to set the default storage engine (replace `InnoDB` with your preferred engine):

```ini
[mysqld]
default_storage_engine=InnoDB
```

Save the file and restart MariaDB:

```bash
sudo systemctl restart mariadb
```

#### **Step 4: Convert Tables to Preferred Storage Engine**

For existing tables, you may need to convert them to your preferred storage engine. Use the `ALTER TABLE` statement to change the storage engine:

```sql
ALTER TABLE your_table ENGINE=InnoDB;
```

Replace `your_table` with the name of your table and `InnoDB` with your preferred storage engine.

#### **Conclusion:**
Selecting and using appropriate MariaDB storage engines can significantly impact database performance and efficiency. Carefully assess the requirements of your application and the characteristics of each storage engine to make informed decisions.

Note: Before making changes to the default storage engine, test the impact on a staging or development environment. Additionally, consider the specific features and limitations of each storage engine for your use case.

## 9. **Regularly Review User Privileges**

**Objective:**
Enhance the security of MariaDB by regularly reviewing and managing user privileges on your Debian system.

#### **Step 1: List Existing Users and Privileges**
Connect to MariaDB and list existing users along with their privileges:
```bash
mysql -u root -p
```

```sql
SHOW GRANTS FOR 'username'@'localhost';
```

Replace `username` with the actual username you want to review. Repeat this process for each user.

#### **Step 2: Remove Unnecessary or Overly Permissive Privileges**
Identify users with unnecessary or overly permissive privileges. Use the `REVOKE` statement to remove privileges:

```sql
REVOKE privilege_type ON database_name.table_name FROM 'username'@'localhost';
```

Replace `privilege_type`, `database_name`, `table_name`, and `username` with the appropriate values.

#### **Step 3: Review and Adjust Global Privileges**
Review and adjust global privileges for users. For example, if a user has unnecessary global privileges, remove them:
```sql
REVOKE SUPER ON *.* FROM 'username'@'localhost';
```

#### **Step 4: Grant Minimum Necessary Privileges**
Ensure that users have the minimum necessary privileges for their tasks. Use the `GRANT` statement to grant specific privileges:
```sql
GRANT SELECT, INSERT, UPDATE ON database_name.table_name TO 'username'@'localhost';
```
Adjust the privileges based on the user's requirements.

#### **Step 5: Remove Unused or Unnecessary User Accounts**
Identify and remove unused or unnecessary user accounts:
```sql
DROP USER 'username'@'localhost';
```
Replace `username` with the actual username you want to remove.

#### **Step 6: Save Changes and Exit**
After making changes, save the changes and exit the MariaDB shell:
```sql
FLUSH PRIVILEGES;
EXIT;
```

#### **Step 7: Schedule Regular Privilege Reviews**
Schedule regular reviews of user privileges to ensure that they remain aligned with security policies. Consider creating a script that automates the privilege review process.

#### **Conclusion:**
Regularly reviewing and managing user privileges in MariaDB is critical for maintaining a secure and well-controlled database environment. By granting the minimum necessary privileges and removing unnecessary access, you reduce the risk of unauthorized actions and potential security vulnerabilities.

Note: Be cautious when revoking privileges to avoid disrupting legitimate operations. Always test changes in a controlled environment before applying them to a production database.

### **Implement Rate Limiting**
**Objective:**
Enhance the security and performance of MariaDB by implementing rate limiting to control the frequency of queries on your Debian system.

#### **Step 1: Install and Configure MariaDB Plugin**
1. Install the `query_response_time` plugin for MariaDB:
   ```bash
   sudo apt-get install mariadb-plugin-query-response-time
   ```
2. Open the MariaDB configuration file:
   ```bash
   sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
   ```
3. Add the following lines to enable the `query_response_time` plugin:
   ```ini
   [mysqld]
   query_response_time_stats = ON
   query_response_time_range = 100-5000
   query_response_time_limit = 5
   ```

   Adjust the `query_response_time_range` and `query_response_time_limit` values based on your specific requirements.

4. Save the file and restart MariaDB:
   ```bash
   sudo systemctl restart mariadb
   ```

#### **Step 2: Monitor and Analyze Query Response Time**
1. Connect to MariaDB:
   ```bash
   mysql -u root -p
   ```
2. Check the current status of the `query_response_time` plugin:
   ```sql
   SHOW VARIABLES LIKE 'query_response_time%';
   ```
3. Monitor and analyze the query response time:
   ```sql
   SELECT * FROM performance_schema.query_response_time;
   ```

   This query provides information about query response times.

#### **Step 3: Adjust Rate Limiting Parameters**
Fine-tune the rate limiting parameters based on the observed query response times and the desired rate limiting strategy. Adjust the `query_response_time_range` and `query_response_time_limit` values accordingly.

#### **Step 4: Monitor and Review Performance Impact**
Regularly monitor the impact of rate limiting on database performance. Use the query response time information to identify potential issues and adjust the rate limiting parameters if needed.

#### **Conclusion:**
Implementing rate limiting in MariaDB helps control the frequency of queries and protects the database from potential abuse or excessive load. Fine-tune the rate limiting parameters based on the specific requirements and characteristics of your application.

Note: Carefully assess the impact of rate limiting on your database performance. Regularly review and adjust the rate limiting parameters to ensure optimal protection without hindering legitimate operations.
